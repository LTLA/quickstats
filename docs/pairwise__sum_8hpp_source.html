56Output_ pairwise_sum(const std::size_t num_total, const Input_* const ptr, Modifier_ mod, PairwiseSumWorkspace<Output_>& work) {
57 work.states.clear();
58
59 std::size_t start = 0, end = num_total;
60 Output_ out = 0;
61 while (1) {
62 const std::size_t len = end - start;
63 if (len > limit_) {
64 work.states.emplace_back(end);
65 end = start + len / 2;
66 continue;
67 }
68

72 }
73
74 start = end;

83 break;
84 }

88 }

118/* COMMENTS:
119 * I tried to write a multi-threaded version of this where each direct summation was submitted to a separate worker until all workers were occupied,
120 * and then added the results once they became available from each worker.
121 * This worksharing is fine-grained but imposes a high cost for inter-thread communication relative to the summation for small `limit_`.
122 * As a consequence, the performance of this multi-threaded version is worse than its serial counterpart.
123 *
124 * I could have implemented alternative approaches that involve less communication but require more memory.
125 * For example, we could split elements into the subarrays ahead of time, distribute the summations to threads once, and then sum the results once all workers are done.
126 * This greatly reduces the cross-talk between threads but requires an extra allocation to store the results.
127 *
128 * TBH, the easiest and most performant approach to parallelization is to just split your input array into one subarray per worker,
129 * perform the sum within each worker for that subarray, and then add the sums afterwards.
130 * This won't give exactly the same result as serial execution but we've crossed that bridge already.
131 * (If exact results are required, we can split it into ceil(log2(num_workers)) subarrays,
132 * which allows us to follow the same halving as pairwise_sum() to get the exact same result at the cost of suboptimal worksharing.)
133 *
134 * In any case, summation is already so fast that I don't think we need to spend a lot of effort in thinking about parallelization.
135 * Especially given that, in real applications, the other threads will typically be occupied elsewhere.
136 * Indeed, we don't deal with parallelization in other parts of this library, so it would be odd to implement it here.
137 *
138 * I also have a sneaking suspicion that the serial code is already pseudo-parallelized via out-of-order execution,
139 * where the next summation starts before the first one has ended.
140 * I say this because pairwise_sum() somehow manages to be slightly faster than std::accumulate() in our R bindings.
141 */
